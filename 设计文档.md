# 棋牌计分小程序需求文档（V2）

## 1. 目标与范围

本项目是一个微信小程序形态的棋牌计分工具。
核心目标：
- 用户可快速创建或加入房间开始对局
- 对局内仅做计分，不内置复杂棋牌规则
- 服务端持久化用户与历史对局数据

MVP 明确不做：
- 自动判定胜负规则
- 复杂积分算法模板
- 强社交系统（好友、群关系链）

## 2. 业务流程

### 2.1 开始界面
每个用户进入首页后有两个入口：
1. 创建房间：服务端生成 5 位房间码（字母+数字）
2. 通过房间码加入房间

房间码规则（初版）：
- 长度：5
- 字符集：`A-Z` + `0-9`
- 示例：`A9K2M`
- 生成策略：随机生成 + 数据库唯一索引兜底重试

### 2.2 对局内流程
1. 房主创建对局并等待玩家加入
2. 所有玩家进入后开始计分
3. 每回合由玩家手动输入点数,只允许支出
4. 系统只做累计展示，不做规则校验
5. 对局结束后写入历史记录

说明：
- “互相支出”由玩家线下约定并手动录入
- 小程序仅保证数据记录、汇总和可追溯

## 3. 用户身份与登录

登录方式：微信小程序身份登录。

基础流程：
1. 小程序端调用 `wx.login()` 获取 `code`
2. 前端将 `code` 发给后端
3. 后端调用微信 `code2session` 接口换取 `openid`（及 `session_key`）
4. 后端以 `openid` 作为用户唯一标识
5. 用户首次登录创建 `User`，后续登录更新必要信息

用户资料：
- 头像 `avatarUrl`
- 昵称 `nickname`
- 平台唯一标识 `openid`

## 4. 数据模型设计（MongoDB）

> 局外需要存用户历史，因此“对局”独立建模，`User` 仅维护最近历史引用（最多 10 局）。

### 4.1 User（局外用户模型）

建议集合：`users`

字段：
- `_id`
- `openid`（唯一索引）
- `nickname`
- `avatarUrl`
- `recentMatchIds`：最近历史对局 ID 列表（最多 10 条）
- `createdAt`
- `updatedAt`

约束：
- `recentMatchIds` 超过 10 条时保留最新 10 条

### 4.2 Match（对局模型）

建议集合：`matches`

字段：
- `_id`
- `roomCode`（创建时绑定，便于追溯）
- `status`：`waiting` | `playing` | `finished`
- `createdBy`（用户 ID）
- `players`：参与用户快照数组
- `rounds`：回合计分记录数组
- `totalScores`：每位玩家累计分
- `startedAt`
- `endedAt`
- `createdAt`
- `updatedAt`

`players` 建议字段：
- `userId`
- `nicknameSnapshot`
- `avatarSnapshot`

`rounds` 建议字段：
- `roundNo`
- `scores`（`userId -> score`）
- `operatorUserId`
- `createdAt`

说明：
- 使用快照是为了避免用户后续改昵称影响历史对局展示

### 4.3 Room（房间模型，可与 Match 合并）

建议集合：`rooms`

字段：
- `_id`
- `roomCode`（唯一索引）
- `hostUserId`
- `memberUserIds`
- `currentMatchId`（可为空）
- `status`：`open` | `closed`
- `expireAt`（可选，便于过期清理）

## 5. 接口草案

### 5.1 认证与用户
- `POST /api/auth/wx-login`：code 登录，返回 token 与用户信息
- `GET /api/users/me`：获取当前用户资料

### 5.2 房间
- `POST /api/rooms`：创建房间并返回 roomCode
- `POST /api/rooms/join`：通过 roomCode 加入房间
- `GET /api/rooms/{roomCode}`：查询房间信息

### 5.3 对局与计分
- `POST /api/matches`：基于 roomCode 创建对局
- `POST /api/matches/{matchId}/rounds`：新增回合分数
- `PATCH /api/matches/{matchId}/rounds/{roundNo}`：修改回合分数
- `POST /api/matches/{matchId}/finish`：结束对局
- `GET /api/matches/history`：获取我的历史对局（分页）

## 6. 技术选型与适配评估

当前选型：
- 后端：Go
- 数据库：MongoDB
- 前端：React.js（编译到微信小程序）

### 6.1 Go 是否适配微信小程序接口要求

结论：适配，且是可行方案。

原因：
- 微信小程序只要求后端提供合法 HTTPS 接口，语言不限
- Go 在高并发、低延迟、部署成本方面适合房间和计分场景
- Go 调用微信开放接口（如 `code2session`）实现简单稳定

实现注意项：
- 服务端必须 HTTPS，且域名配置到小程序后台白名单
- 登录链路按 `wx.login -> code2session -> openid` 执行
- 接口统一 JSON，建议使用 JWT 或会话 token
- 关键索引：`users.openid`、`rooms.roomCode`
- 需要防止 roomCode 冲突和重复提交导致的脏写

### 6.2 React.js 在微信小程序中的落地方式

说明：微信小程序原生不是 React 运行时，React 需通过跨端框架编译。

建议：
- 使用 Taro（React 语法）编译到微信小程序
- 或使用 uni-app（Vue 体系，不是 React）

若坚持 React.js，推荐技术路线：`React + Taro + Go + MongoDB`。

## 7. 非功能要求

- 数据可靠：对局写入成功后才返回成功状态
- 并发安全：同一回合避免重复提交（幂等处理）
- 性能目标：单房间常规人数（2-8）操作无明显卡顿
- 可维护性：对局与用户解耦建模，便于后续扩展规则系统

## 8. MVP 验收标准

1. 用户可通过微信身份登录并成功拉取头像昵称
2. 用户可创建 5 位房间码房间并被他人加入
3. 房间内可创建对局并完成至少 20 回合计分
4. 系统正确累计总分并可修改历史回合
5. 对局结束后写入历史，用户侧最多保留最近 10 局引用

## 9. 待定项（后续细化）

- 房间人数上限（默认建议 8）
- 房间有效期（是否 24 小时自动过期）
- 是否支持观战模式
- 是否需要对局详情导出分享图
- 回合修改权限（仅房主 / 全员可改）
